# 3.1.2.research.external.factory.oss.levers._.v1

research on open source tools and libraries available to build the patenter role.

---

## summary

the patenter role can leverage a rich ecosystem of open source tools across several categories: patent API clients, PDF/XML parsing, search engines, CLI frameworks, and template systems. key findings:

1. **patent API clients are fragmented** — the unified `patent_client` library is archived; separate clients exist for USPTO and EPO
2. **ip-tools org is authoritative** — maintains both `python-epo-ops-client` and `uspto-opendata-python`
3. **Google Patents has no official API** — must use SerpApi (paid) or scrapers (ToS risk)
4. **Typer + Rich is the modern Python CLI standard** — type hints, beautiful output
5. **LaTeX patent templates are scarce** — will need custom templates
6. **vector search enables semantic prior art matching** — Chroma and Qdrant for embeddings

## conflict detection

### patent_client vs individual clients

sources [1] (patent_client) provided a unified interface but is now archived. sources [2], [3], [4] provide individual clients for each patent office. **recommendation**: use individual clients (ip-tools) rather than the archived unified solution.

### Google Patents access

source [5] (SerpApi) provides official paid API access. sources [6], [7] provide scraping alternatives. **conflict**: paid vs free with ToS risk. **recommendation**: use SerpApi for production if budget allows; use scraping only for development/testing with caution.

### PDF extraction: speed vs detail

source [10] (PyMuPDF) emphasizes speed. source [11] (pdfplumber) emphasizes detail (tables, char positions). **recommendation**: use PyMuPDF for simple text extraction, pdfplumber for structured extraction (tables, claims).

### search: embedded vs service

source [18] (Whoosh) is embedded/pure-Python. sources [16], [17], [19] require separate services. **recommendation**: use Whoosh for development/small scale; Meilisearch or Typesense for production.

## convergence signals

### ip-tools org is authoritative (2 sources agree)

sources [2] and [3] both from ip-tools org maintain consistent patterns across EPO and USPTO clients. this is the de facto standard for Python patent API access.

### Typer + Rich is the Python CLI standard (3+ sources agree)

sources [23], [24], [25] together form the modern Python CLI stack. Typer builds on Click, Rich provides output formatting. many new CLI projects use this combination.

### vector DBs for semantic search (3+ sources agree)

sources [20], [21], [32] converge on embedding-based search for semantic similarity. this enables "find patents similar to my invention" queries beyond keyword matching.

### LaTeX templates are lacking (search convergence)

no actively maintained USPTO-specific LaTeX template was found across sources. this represents a gap that the patenter role must fill.

## anti-patterns

### do not rely on archived libraries

> "patent_client is archived and should not be used as a primary dependency." [1]

archived libraries will not receive security patches or API updates. use actively maintained alternatives.

### do not scrape without fallback

> "Scraping websites like Google Patents may breach their terms of service." [from prior research]

sources [6], [7] work today but may break. always have a fallback (official API or bulk data).

### do not embed search for scale

> "Whoosh is not optimized for very large graphs." [18]

source [18] works for development but will not scale to millions of patents. plan for service-based search (Meilisearch, Elasticsearch) in production.

### do not ignore bulk data

> "For high-volume needs with USPTO data, use bulkdata.uspto.gov to download XML files rather than scraping the search GUI." [from prior research]

API rate limits constrain real-time search. bulk downloads provide complete, unrestricted access for indexing.

## recommendations for patenter

### tier 1: core dependencies

| purpose | recommended | rationale |
|---------|-------------|-----------|
| USPTO API | `uspto-opendata-python` [3] | actively maintained, comprehensive |
| EPO API | `python-epo-ops-client` [2] | official API, well-documented |
| PDF extraction | `PyMuPDF` [10] | fast, comprehensive |
| XML parsing | `lxml` [13] | industry standard |
| CLI framework | bash skills | consistent with rhachet patterns |
| output formatting | bash + turtle vibes | consistent with mechanic role |

### tier 2: enhanced capabilities

| purpose | recommended | rationale |
|---------|-------------|-----------|
| Google Patents | `SerpApi` [5] if budget allows | official API |
| semantic search | `Chroma` [20] + `sentence-transformers` [32] | prior art similarity |
| full-text search | `Meilisearch` [16] | fast, typo-tolerant |
| citation networks | `NetworkX` [22] | graph analysis |

### tier 3: build custom

| purpose | approach | rationale |
|---------|----------|-----------|
| LaTeX templates | custom templates | no good OSS exists |
| route system | bhrain integration | consistent with rhachet |
| claim parsing | custom NLP pipeline | domain-specific structure |
