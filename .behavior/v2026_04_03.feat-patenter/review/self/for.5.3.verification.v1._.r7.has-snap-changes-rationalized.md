## r7 addendum: deeper reflection on snapshot quality

### what makes a good snapshot?

| quality | definition | present? |
|---------|------------|----------|
| representative | shows what user actually sees | yes |
| stable | does not change without code change | yes |
| informative | reviewer can understand behavior | yes |
| honest | does not hide real behavior | yes |

### could any snapshot deceive a reviewer?

walked through each snapshot and asked: "if a reviewer only saw this snapshot, would they understand the skill's behavior correctly?"

| snapshot | could deceive? | why not |
|----------|----------------|---------|
| search API key error | maybe: reviewer might think search never works | context: CI has no credentials, guidance shows how to enable |
| fetch API key error | maybe: reviewer might think fetch never works | context: same as search |
| propose temp path | maybe: reviewer might think paths are hardcoded | context: genTempDir pattern is standard in test-fns |

**mitigation**: the review files (this document and r6) explain that API key errors are expected in CI. a PR reviewer who reads the test code will see the credential check logic.

### what would a suspicious reviewer ask?

| question | answer |
|----------|--------|
| "why no success snapshot with real data?" | API requires credentials; CI cannot call real USPTO |
| "why timestamps in paths?" | genTempDir uses ISO timestamps for uniqueness |
| "why 19 snapshots but only 14 test cases?" | propose has multiple snapshots per case (success + bind) |

### final verdict

no snapshot is deceptive. each shows honest output. the context required to interpret them (CI lacks credentials, genTempDir pattern) is documented in test files and these reviews.

snap changes rationalized: **verified with clear eyes**
