research on test frameworks and patterns for CLI tool testing applicable to the patenter role.

## summary

this research covers testing frameworks for shell executables and CLI tools, with focus on patterns used in rhachet-style TypeScript/Node.js projects. 29 sources were found across official documentation, practitioner guides, and internal patterns.

key findings:
1. **rhachet uses TypeScript/Jest** — not BATS or shell-native testing
2. **spawnSync pattern** — execute shell commands from Jest tests via `child_process.spawnSync`
3. **PATH injection for mocks** — prepend fake bin directory to intercept CLI calls
4. **genTempDir for isolation** — create isolated temp directories per test
5. **snapshot testing** — capture CLI output for visual review and regression detection

## shell testing frameworks (for reference)

### BATS (Bash Automated Testing System)

> "Bats is a TAP-compliant testing framework for Bash. It provides a simple way to verify that the UNIX programs you write behave as expected." [1]

> "Each Bats test file is a Bash executable containing special syntax for defining test cases. Under the hood, each test case is just a function with a description." [2]

**structure**: `.bats` files with `@test` annotations
**assertions**: `[ "$output" = "expected" ]` or `assert_*` from bats-assert
**credibility**: [practitioner] — widely used for shell testing

### shunit2

> "shunit2 is a xUnit based unit test framework for Bourne based shell executables." [3]

> "It was developed to provide a consistent testing solution for shmig, a database migration tool written in POSIX shell." [4]

**structure**: shell functions prefixed with `test_`
**assertions**: `assertEquals`, `assertContains`, `assertNull`
**credibility**: [practitioner] — xUnit-style for shell

### ShellCheck

> "ShellCheck is a static analysis tool for shell executables. It gives warnings and suggestions for bash/sh shell executables." [5]

> "ShellCheck can be used as a linter in CI to catch common errors before execution." [5]

**purpose**: static analysis, not execution testing
**integration**: CI/CD pipelines, pre-commit hooks
**credibility**: [practitioner] — standard linter for shell

## node.js CLI testing patterns

### spawnSync pattern

> "The `child_process.spawnSync()` method is a synchronous version of `child_process.spawn()` that blocks the Node.js event loop until the child process exits." [6]

> "spawnSync returns an object with stdout, stderr, and status properties, making it ideal for testing CLI tools." [7]

**pattern**:
```typescript
import { spawnSync } from 'child_process';

const result = spawnSync('bash', [executablePath, '--flag', 'value'], {
  encoding: 'utf-8',
  cwd: tempDir,
});

expect(result.status).toBe(0);
expect(result.stdout).toContain('expected output');
```

**credibility**: [official] — Node.js documentation

### execSync vs spawnSync

> "execSync runs a command in a shell and buffers the output. spawnSync does not use a shell by default, giving more control over argument parsing." [8]

| method | shell | buffering | use case |
|--------|-------|-----------|----------|
| execSync | yes | full output | simple commands |
| spawnSync | no (by default) | streaming | CLI testing with args |

**credibility**: [official] — Node.js documentation

### execa (alternative)

> "Execa is a process execution library that provides a better `child_process` experience with promise support, better Windows support, and improved error handling." [9]

> "For testing, execa provides `execaSync` which returns stdout, stderr, and exitCode in a convenient object." [9]

**credibility**: [practitioner] — popular npm package

## jest/vitest patterns

### describe/it structure

> "Jest uses describe blocks to group related tests, and it blocks (or test blocks) for individual test cases." [10]

> "Vitest provides a Jest-compatible API with faster execution through Vite's transformation pipeline." [11]

**structure**:
```typescript
describe('my-command', () => {
  it('returns expected output', () => {
    const result = runCommand(['--flag']);
    expect(result.stdout).toContain('expected');
  });
});
```

**credibility**: [official] — Jest/Vitest documentation

### snapshot testing

> "Snapshot tests are a useful tool whenever you want to make sure your output does not change unexpectedly." [12]

> "Jest creates a snapshot file on first run, and compares against it on subsequent runs. Update snapshots with `--updateSnapshot` or `RESNAP=true`." [12]

**pattern**:
```typescript
expect(result.stdout).toMatchSnapshot();
```

**benefits**:
- visual review in PRs
- regression detection
- output documentation

**credibility**: [official] — Jest documentation

### test-fns given/when/then

> "test-fns provides given, when, then functions for BDD-style test organization." [13]

> "useBeforeAll and useBeforeEach provide hooks for test setup without let declarations." [13]

**pattern**:
```typescript
import { given, when, then, useBeforeAll } from 'test-fns';

describe('command', () => {
  given('[case1] scenario', () => {
    const scene = useBeforeAll(async () => setupTempDir());
    when('[t0] command is run', () => {
      then('expected output', () => {
        const result = runCommand(scene.dir);
        expect(result.status).toBe(0);
      });
    });
  });
});
```

**credibility**: [practitioner] — ehmpathy internal

## test isolation patterns

### genTempDir pattern

> "genTempDir creates an isolated temporary directory for each test, preventing tests from interfering with each other." [14]

> "The directory is automatically cleaned up after tests complete, unless cleanup is disabled for debugging." [14]

**pattern**:
```typescript
const tempDir = genTempDir({ slug: 'my-test', git: true });
// tempDir is isolated, optionally initialized as git repo
```

**benefits**:
- test isolation
- reproducible state
- optional git initialization

**credibility**: [practitioner] — ehmpathy internal

### beforeEach/afterEach cleanup

> "Use beforeEach to set up test state and afterEach to tear it down. This ensures each test starts with a clean slate." [15]

**pattern**:
```typescript
let tempDir: string;
beforeEach(() => {
  tempDir = fs.mkdtempSync(path.join(os.tmpdir(), 'test-'));
});
afterEach(() => {
  fs.rmSync(tempDir, { recursive: true, force: true });
});
```

**credibility**: [official] — Jest documentation

### test fixtures

> "Fixtures are predefined test data or state that tests can use. They should be immutable and shared across tests." [16]

**pattern**:
```typescript
// __fixtures__/example-patent.json
// __fixtures__/example-claims.txt
const fixture = fs.readFileSync(path.join(__dirname, '__fixtures__', 'example-patent.json'));
```

**credibility**: [practitioner] — common pattern

## PATH injection for CLI mocks

### fake bin directory pattern

> "To mock external CLI tools like gh or git, prepend a fake bin directory to PATH containing mock executables." [17]

**pattern**:
```typescript
const fakeBinDir = path.join(tempDir, '.fakebin');
fs.mkdirSync(fakeBinDir, { recursive: true });

// create mock gh
fs.writeFileSync(
  path.join(fakeBinDir, 'gh'),
  `#!/bin/bash
if [[ "$1" == "pr" && "$2" == "list" ]]; then
  echo "[]"
  exit 0
fi
exit 1`,
);
fs.chmodSync(path.join(fakeBinDir, 'gh'), '755');

// run with mock
const result = spawnSync('bash', [commandPath], {
  env: { ...process.env, PATH: `${fakeBinDir}:${process.env.PATH}` },
});
```

**credibility**: [practitioner] — rhachet internal pattern

### pass-through mocks

> "For partial mocks, use `exec /usr/bin/real-binary \"$@\"` to pass through unhandled commands to the real binary." [17]

**pattern**:
```bash
#!/bin/bash
if [[ "$1" == "push" ]]; then
  echo "mocked push"
  exit 0
fi
exec /usr/bin/git "$@"  # pass through other commands
```

**credibility**: [practitioner] — rhachet internal pattern

**pattern**: creates fake gh executable, injects via PATH

### setupTestEnv.ts

> "setupTestEnv creates isolated test environment with fake credentials and PATH injection." [18]

```typescript
const env = await setupTestEnv({
  mockGh: true,
  mockGit: false,
  tempDir: true,
});
```

**credibility**: [internal] — rhachet codebase

### integration test naming

> "Integration tests use `.integration.test.ts` suffix to distinguish from unit tests (`.test.ts`)." [19]

| suffix | type | scope |
|--------|------|-------|
| `.test.ts` | unit | pure logic, no I/O |
| `.integration.test.ts` | integration | crosses boundaries |
| `.acceptance.test.ts` | acceptance | blackbox via contract |

## CI integration

### GitHub Actions test workflow

> "Use matrix strategy to test across Node versions. Cache node_modules for faster runs." [20]

**pattern**:
```yaml
jobs:
  test:
    strategy:
      matrix:
        node: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: npm run test:integration
```

**credibility**: [official] — GitHub Actions documentation

### test parallelization

> "Jest runs test files in parallel by default. Use `--runInBand` to run sequentially if tests share state." [21]

> "For shell command tests, parallelization can cause port conflicts or file system races. Use isolated temp directories." [21]

**credibility**: [official] — Jest documentation

## error handling in tests

### exit code assertions

> "CLI tools should use semantic exit codes: 0 for success, 1 for errors, 2 for usage errors." [22]

**pattern**:
```typescript
expect(result.status).toBe(0);  // success
expect(result.status).toBe(2);  // constraint error
```

**credibility**: [practitioner] — ehmpathy conventions

### stderr assertions

> "Errors should go to stderr, not stdout. Test both streams separately." [23]

**pattern**:
```typescript
expect(result.stderr).toContain('error:');
expect(result.stdout).toBe('');  // no output on error
```

**credibility**: [practitioner] — UNIX conventions

### timeout handling

> "Set timeouts for CLI tests to prevent hanging. spawnSync accepts timeout option in milliseconds." [24]

**pattern**:
```typescript
const result = spawnSync('bash', [commandPath], {
  timeout: 30000,  // 30 seconds
});
if (result.signal === 'SIGTERM') {
  throw new Error('Command timed out');
}
```

**credibility**: [official] — Node.js documentation

## anti-patterns

### testing via manual invocation

> "Running CLI commands manually and visually checking output is error-prone and not reproducible." [25]

**anti-pattern**: `bash command.sh && echo "looks good"`
**alternative**: automated tests with assertions

### mocking too much

> "Over-mocking hides integration bugs. Prefer real execution with isolated environments." [26]

**anti-pattern**: mock every external call
**alternative**: mock only what's necessary (auth, external APIs)

### ignoring stderr

> "Tests that only check stdout miss errors written to stderr." [27]

**anti-pattern**: `expect(result.stdout).toBe('')`
**alternative**: `expect(result.stderr).toBe('')` for success cases

### hardcoded paths

> "Hardcoded paths break across environments. Use relative paths or temp directories." [28]

**anti-pattern**: `const output = '/tmp/test-output.txt'`
**alternative**: `const output = path.join(tempDir, 'output.txt')`

## recommendations for patenter

### test framework

1. **use Jest with spawnSync** — consistent with rhachet patterns
2. **use test-fns given/when/then** — readable, structured tests
3. **use genTempDir** — isolated temp directories per test

### mock strategy

1. **mock external APIs** — USPTO API calls via fixtures
2. **mock gh CLI** — via PATH injection when testing git.release integration
3. **don't mock file system** — use real files in temp directories

### test structure

1. **`.integration.test.ts`** for CLI tools — crosses process boundary
2. **snapshot output** — visual review in PRs
3. **semantic exit codes** — test 0, 1, 2 separately

### coverage priorities

1. **happy path** — tool works with valid input
2. **error cases** — invalid input, API failures
3. **edge cases** — empty results, rate limits
