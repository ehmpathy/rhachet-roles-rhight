opportunities analysis for the patenter role implementation.

## summary

| opportunity | impact | effort | recommendation |
|-------------|--------|--------|----------------|
| patent document cache | high | medium | **build** — enables fast reruns |
| parallel search queries | medium | low | **defer** — optimize later |
| fixture generation tool | medium | low | **build** — speeds test creation |

**the bottleneck**: API round-trip time for patent search and fetch operations.

**the opportunity**: local patent document cache. patents are immutable after publication — cache once, reuse forever.

---

## bottleneck analysis

### where time goes

| phase | time | cacheable |
|-------|------|-----------|
| query formulation | 1s | no |
| API request (search) | 2-5s | partial (same query) |
| API request (fetch) | 3-8s | **yes** (patent immutable) |
| parse response | <1s | no |
| write output | <1s | no |

**70% of verification time is network wait** — and fetch results are perfectly cacheable.

### why patent fetch is special

unlike most API data:
- patents are **immutable after publication**
- same exid always returns same content
- no freshness concerns (no TTL needed)
- perfect cache candidate

## opportunity.1 = patent document cache

### what

local file cache for fetched patent documents.

```
~/.cache/rhachet/patents/
├── US20210234567A1.json
├── US20200123456A1.json
└── ...
```

### why high impact

| scenario | without cache | with cache |
|----------|---------------|------------|
| first fetch | 5s | 5s |
| second fetch (same patent) | 5s | <100ms |
| route re-run | 30s (6 patents) | <1s |
| test suite | 60s (12 fetches) | <1s |

### implementation sketch

```bash
# in patent.priors.fetch
CACHE_DIR="${XDG_CACHE_HOME:-$HOME/.cache}/rhachet/patents"
CACHE_FILE="$CACHE_DIR/$EXID.json"

# check cache first
if [[ -f "$CACHE_FILE" ]]; then
  cat "$CACHE_FILE"
  exit 0
fi

# fetch from API
RESULT=$(curl -s "$API_URL/$EXID" ...)

# cache result
mkdir -p "$CACHE_DIR"
echo "$RESULT" > "$CACHE_FILE"

# output
echo "$RESULT"
```

### effort estimate

- implementation: 20 lines of bash
- test: verify cache hit/miss behavior
- total: 1-2 hours

### recommendation

**build** — high impact, low effort, enables fast iteration on routes.

## opportunity.2 = parallel search queries

### what

run multiple search queries concurrently when research phase needs varied perspectives.

### why medium impact

research phase often runs 3-5 searches with different query phrasings:
- sequential: 15-25s
- parallel: 5-8s

### implementation sketch

```bash
# run searches in background
search_query1 &
search_query2 &
search_query3 &
wait
```

### effort estimate

- implementation: bash background jobs
- complexity: merge results, handle failures
- total: 2-3 hours

### recommendation

**defer** — nice optimization, but not critical for v1. sequential is acceptable.

## opportunity.3 = fixture generation tool

### what

command to capture live API responses as test fixtures.

```bash
rhx patent.fixtures.capture --exid US20210234567A1 --into __fixtures__/
```

### why medium impact

| approach | time to create fixture |
|----------|------------------------|
| manual copy-paste | 5-10 min |
| fixture capture tool | 30s |

with 10+ fixtures needed, saves 1-2 hours.

### implementation sketch

```bash
# capture search response
curl -s "$SEARCH_API?q=$QUERY" | jq '.' > __fixtures__/search-$SLUG.json

# capture fetch response
curl -s "$FETCH_API/$EXID" | jq '.' > __fixtures__/patent-$EXID.json
```

### effort estimate

- implementation: simple curl wrapper
- total: 30 min

### recommendation

**build** — low effort, enables fast test authorship.

## opportunity.4 = shared API client

### what

extract API call logic into shared operations file.

### why low impact

- curl + jq is already simple
- bash skills are small (50-100 lines each)
- abstraction adds indirection without significant reuse

### recommendation

**skip** — YAGNI. inline curl is fine for 3 skills.
