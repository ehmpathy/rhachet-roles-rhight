## premortem: imagine we shipped and it failed

project forward: we shipped patenter, and it failed badly. why?

### failure scenario 1: API access breaks

USPTO changes their API, adds authentication, or rate limits us into uselessness. the skill returns empty results or errors. users can't search or fetch patents.

**likelihood**: medium
**impact**: complete feature outage

### failure scenario 2: search results are poor

users search for their invention and get irrelevant patents. they miss critical prior art that later invalidates their claims. or they get flooded with 500 results and can't find what matters.

**likelihood**: medium
**impact**: false confidence, wasted attorney spend, rejected applications

### failure scenario 3: users abandon the route

the stone workflow is unfamiliar. users fill out 0.idea.md, get confused at 3.1, and give up. the route feels too long or too bureaucratic. adoption flatlines.

**likelihood**: medium
**impact**: tool becomes shelfware

### failure scenario 4: claims output is garbage

the 3.2 distillation produces claims that make no sense. attorneys laugh at the output. users waste time and lose trust.

**likelihood**: low-medium (depends on briefs quality)
**impact**: reputation damage, abandoned tool

### failure scenario 5: legal liability exposure

a user treats patenter output as legal advice. they file a bad application based on our analysis. they get rejected or, worse, sued for infringement. they blame us.

**likelihood**: low (with disclaimers)
**impact**: severe — legal and reputation risk

---

## what obvious signs did we ignore?

signs we might dismiss but shouldn't:

- **"the API seems unreliable"** — occasional 500s, slow responses, rate limit bumps
- **"users don't read the briefs"** — they skip the terminology docs and get confused
- **"the route is too long"** — 8 stones feels heavy for an MVP
- **"search results vary wildly"** — same query returns different results on different days
- **"no one finishes the route"** — users start but don't complete

---

## what assumptions could turn out wrong?

| assumption | what if wrong? | likelihood |
|------------|----------------|------------|
| USPTO ODP API is stable and accessible | feature outage, need alternative API | medium |
| inventors can describe ideas clearly | garbage in, garbage out | medium |
| CLI is acceptable for target audience | need GUI, wrong audience | low |
| patent search is tractable with keywords | need semantic search, ML-based rank | medium |
| route workflow is learnable | users bounce, need intro guide | medium |
| latex output is attorney-acceptable | need DOCX, different format | low (2024 rule requires DOCX anyway) |
| claims draft adds value | attorneys ignore it, users confused | medium |
| bhrain route system is stable | integration breaks, route fails | low |

---

## what is the nightmare scenario?

**the worst plausible outcome**:

a user relies on patenter to research prior art for their startup's core IP. they file a provisional based on our analysis. they raise capital on the IP position. a competitor finds prior art we missed. the patent is invalidated. the startup loses its IP moat. the user blames patenter publicly. our reputation is destroyed.

**what makes this nightmare**:
- **false confidence is worse than no confidence** — we gave them a green light that was actually red
- **financial stakes are high** — startup capital, IP value, legal fees
- **recovery is hard** — invalidated patents can't be un-invalidated
- **public blame spreads** — one angry user on Twitter/HN damages adoption

---

## what mitigations to consider?

| risk | mitigation | cost |
|------|------------|------|
| API breaks | multiple API sources (ODP + PatentsView + Google Patents) | medium — 2x integration |
| poor search results | relevance score + "try these alternative queries" | low — query refinement prompts |
| users abandon route | shorter MVP route (5 stones), progress indicators | low — template simplification |
| claims output garbage | conservative tone ("draft for review, not final") | low — disclaimer + tone |
| legal liability | explicit "not legal advice" disclaimers everywhere | low — copy changes |
| missed prior art | multiple search passes, "did you search for X?" prompts | low — process change |
| false confidence | surface uncertainty ("3 searches done, 47 results reviewed") | low — transparency |

---

## key mitigations to implement

### must-have (before ship)

1. **"not legal advice" disclaimer** — prominent in output and briefs
2. **multiple search passes** — encourage varied queries, surface count
3. **conservative claims tone** — "draft for attorney review"
4. **API fallback plan** — document what to do if API breaks

### should-have (soon after ship)

1. **progress indicators** — show where user is in route
2. **query refinement suggestions** — "also try these terms"
3. **shorter MVP route** — optional fast path for provisional

### nice-to-have (future)

1. **multiple API sources** — ODP + PatentsView + Google
2. **semantic search** — ML-based relevance
3. **GUI companion** — for non-CLI users

---

## summary

**most likely failure**: poor search results that give users false confidence.

**most severe failure**: user relies on our analysis, files bad application, loses IP position, blames us publicly.

**key mitigations**: disclaimers, multiple search passes, conservative tone, API fallback plan.

**action**: implement must-have mitigations before ship. track adoption metrics to catch "users abandon route" early.
