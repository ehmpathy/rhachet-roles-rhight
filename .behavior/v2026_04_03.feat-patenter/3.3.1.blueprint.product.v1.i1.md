```
    │      # --query "$TEXT" --limit N --since YYYY-MM-DD --until YYYY-MM-DD
    │      # extract into: QUERY_TEXT, PAGE_LIMIT (default 20), DATE_SINCE, DATE_UNTIL
    │
    ├─ [+] validate_query()
    │      # if len < 3: exit 2 "query too short"
    │      # if len > 1000: exit 2 "query too long"
    │      # sanitize: trim whitespace, escape special chars for URL
    │      # extract keywords: split on whitespace, filter stopwords
    │
    ├─ [+] search_uspto()
    │      # endpoint: GET https://data.uspto.gov/apis/patent-file-wrapper/search
    │      # headers: X-API-KEY: $USPTO_ODP_API_KEY
    │      # params: q=$QUERY_TEXT (OpenSearch syntax)
    │      #         rows=$PAGE_LIMIT
    │      #         start=0
    │      # if DATE_SINCE: add datePublished:[$DATE_SINCE TO *]
    │      # if DATE_UNTIL: add datePublished:[* TO $DATE_UNTIL]
    │      # curl with 10s timeout
    │      # if 429: read Retry-After header, sleep, retry with exponential backoff + jitter
    │      # if network error: exit 1 "API unavailable: check network connection"
    │
    ├─ [+] parse_results()
    │      # jq: .results[] | {exid:.applicationNumber, title:.title,
    │      #     abstract:.abstract, datePublished:.publicationDate}
    │      # if .results null or empty:
    │      #   set RESULTS=[] and ALERT="no results found"
    │      #   set SUGGESTION="try broader terms, synonyms, or check for typos"
    │      # if query has <3 keywords: set ALERT="query may be too vague"
    │
    ├─ [+] filter_and_score()
    │      # USPTO API returns OR matches (any keyword), so filter for title relevance
    │      # filter: require at least ONE keyword in title (title relevance gate)
    │      # score: keyword matches in title (2x) + abstract (1x)
    │      # scale to 0-1: score / max_score
    │      # sort descending by relevance
    │      # if filter removes all results: set SUGGESTION="no patents have search terms in title"
    │
    ├─ [+] emit_results()
    │      # 🦅 lets soar and see,
    │      # 🌎 patent.priors.search
    │      #    ├─ query: $QUERY_TEXT
    │      #    ├─ results: N patents
    │      #    └─ patents
    │      #       ├─ US20210234567A1 (0.87)
    │      #       │  └─ Neural network compression via pruning
    │      #       └─ ...
    │      # if ALERT: print after results
    │      # if SUGGESTION: print after ALERT
    │
    └─ [←] output.sh
```

**cache structure (hard requirement)**:
- `.cache/patents/{exid}.meta.json` — raw metadata API response (prosecution timeline events, dates)
- `.cache/patents/{exid}.doc.xml` — full granted patent document XML (claims, description, abstract)
- `.cache/patents/{exid}.fig-{n}.{ext}` — figure images (fig-1.png, fig-2.tiff, etc.)
- `.cache/patents/{exid}.docs.json` — prosecution document list from API
- `.cache/patents/{exid}.docs/` — **prosecution documents directory (REQUIRED)**
  - `{docCode}.{date}.pdf` — e.g., `CTFR.2019-03-15.pdf` (final rejection)
  - `{docCode}.{date}.pdf` — e.g., `A....2019-06-10.pdf` (applicant response)
  - ...all prosecution correspondence PDFs

**cache rules**:
- always cache raw responses, never parsed/transformed
- --cache skip bypasses all cached files and forces fresh fetch
- patent data is immutable, so cache never stales
- if document XML unavailable, emit metadata + prosecution docs
- **prosecution documents are REQUIRED** — timeline events alone are INSUFFICIENT

**fetch order**:
1. metadata (required) — prosecution timeline events, grant info, document URI
2. document XML (if URI available) — claims, description, abstract
3. figures (if referenced in document) — image files
4. **prosecution documents (REQUIRED)** — office actions, responses, amendments, notices

## test coverage

### mandatory requirements

1. **`--help` on every skill** — each skill must support `--help` flag that displays usage info
2. **stdout snapshots on every test** — every `then()` block must include `expect(result.stdout).toMatchSnapshot()` for vibecheck in PR reviews
3. **live API calls required** — integration tests MUST hit the real USPTO API for success paths. fixtures are only for constraint/error path tests where we control the input validation. a test that mocks or fixtures the API response for a success path is NOT an integration test.

### integration tests

**critical: live API requirement**

tests for `patent.priors.search` and `patent.priors.fetch` success paths MUST call the live USPTO API. this is non-negotiable. the purpose of integration tests is to verify the real system works end-to-end.

| skill | test file | coverage |
|-------|-----------|----------|
| patent.priors.search | `patent.priors.search.integration.test.ts` | help, query too short, no query, **valid query (LIVE API)** |
| patent.priors.fetch | `patent.priors.fetch.integration.test.ts` | help, no exid, invalid format, **valid exid (LIVE API)** |
| patent.propose | `patent.propose.integration.test.ts` | help, route creation, branch bind, extant route, invalid editor |

**what "LIVE API" means:**
- the test makes a real HTTP request to USPTO
- the test receives a real response with real patent data
- the test verifies the response contains expected fields
- for fetch: the cache at `.cache/patents/` (repo-local) is populated with a real patent JSON file

**fixtures usage (limited):**
- `.fixtures/` files are ONLY to document expected response shapes
- fixtures may be used for error path tests (e.g., to simulate 404 not found)
- fixtures MUST NOT be used for success path tests

**CRITICAL: multiple patent types must work**

the search must return results for different categories of inventions. tests with only one query (e.g., "neural network compression") are insufficient. we must prove the API works for various domains.

**IMPORTANT: USPTO API behavior**

the USPTO search API returns OR matches (any keyword), not AND matches. when a user searches "power drill", the API returns patents with "power" OR "drill" in their text. combined with the fact that patent titles use technical terminology (e.g., "POWERED FASTENER DRIVER" not "power drill"), searches with consumer terms often return irrelevant results.

the search skill filters results to require at least one keyword in the title, then scores by keyword frequency. this provides better relevance than raw API results.

**recommended test queries** (use technical patent terminology):
- `fastener driver` — finds drills, impact drivers, screwdrivers
- `rotary saw` — finds saws, blades
- `wireless transceiver` — finds radio, wireless patents
- `neural network compression` — finds ML patents (original test case)
