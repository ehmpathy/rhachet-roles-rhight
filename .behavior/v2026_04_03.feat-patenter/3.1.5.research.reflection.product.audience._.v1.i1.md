# 3.1.5.research.reflection.product.audience._.v1

audience analysis for the patenter role.

---

## who is the audience?

### primary: technical founders and indie inventors

**profile**:
- comfortable with CLI tools and terminal workflows
- has a novel idea they want to protect
- wants to understand the patent landscape before attorney spend
- capable of technical research but unfamiliar with patent process
- budget-conscious — $15-30k for attorney-led application is painful

**examples**:
- solo founder with a machine learn breakthrough
- indie hacker with a unique API design
- researcher with a publishable algorithm who wants IP first
- CTO at a startup who needs to build a patent portfolio

### secondary: patent attorneys (future scope)

**profile**:
- handles 10-50 patent applications per year
- spends significant time on prior art research
- would delegate research automation if reliable
- values comprehensive search coverage

**not in v1 scope** — focus on inventors first.

### tertiary: technical co-founders and CTOs

**profile**:
- not the inventor, but needs to understand IP position
- reviews patent applications for strategic alignment
- makes build-vs-buy decisions on IP tools

---

## why do they care?

### pain relieved

| pain | intensity | how patenter helps |
|------|-----------|-------------------|
| prior art search is tedious | high | automated search with relevance scores |
| patent language is arcane | high | structured route guides claim draft |
| attorney fees are steep | high | DIY provisional, informed attorney handoff |
| fear of missed prior art | medium | systematic search with multiple queries |
| uncertainty about novelty | high | clear favorable vs adverse categorization |

### goals enabled

| goal | how patenter helps |
|------|-------------------|
| file provisional to establish priority date | route produces draft materials |
| understand patent landscape before full investment | search + fetch + categorization |
| prepare for attorney consultation with research done | structured output for handoff |
| defend IP position with awareness of prior art | adverse patent identification |

### frustrations eliminated

| frustration | resolution |
|-------------|------------|
| "I don't know where to start" | 0.idea.md template with guided prompts |
| "I can't find relevant patents" | automated search with query refinement |
| "I don't understand claim structure" | briefs explain patterns, route guides draft |
| "I don't know if my idea is novel" | 3.2 stones produce patentability analysis |

## how much do they care?

| audience | priority | urgency | stakes |
|----------|----------|---------|--------|
| primary (inventors) | high | varies — pre-publication deadline | $15-30k attorney cost, IP protection |
| secondary (attorneys) | medium | continuous | time efficiency, coverage quality |
| tertiary (CTOs) | low | strategic | portfolio value, competitive position |

**note on urgency**: inventors often face publication deadlines. if they publish before file, they lose international rights. this creates time pressure that increases urgency.

---

## what do they care about?

### success criteria

| criterion | why it matters |
|-----------|----------------|
| **coverage** — did I find all relevant prior art? | missed prior art = weak application |
| **clarity** — do I understand what's novel? | vague novelty = weak claims |
| **defensibility** — will claims survive examination? | rejected application = wasted effort |
| **cost** — did I reduce attorney spend? | $15k saved = significant for indie |
| **speed** — can I establish priority date fast? | time to file matters for IP strategy |

### tradeoffs they accept

| tradeoff | accepted? | why |
|----------|-----------|-----|
| automation over perfection | yes | good enough research beats no research |
| CLI over GUI | yes (primary) | primary audience is CLI-native |
| USPTO only over international | yes (v1) | most start with USPTO, expand later |
| utility patents only | yes (v1) | utility is the common case |

### tradeoffs they reject

| tradeoff | rejected? | why |
|----------|-----------|-----|
| inaccurate search results | yes | false confidence is worse than no confidence |
| claims that don't make sense | yes | garbage output wastes time |
| slow search (>30s) | maybe | tolerable if results are good |
| complex setup process | yes | want to start research immediately |

### delight vs disappointment

| scenario | emotion |
|----------|---------|
| "I found a patent I would have missed" | delighted |
| "I now understand why my idea is novel" | delighted |
| "my attorney said I did better research than most clients" | delighted |
| "the search missed an obvious patent" | disappointed |
| "the claims made no sense" | disappointed |
| "the route was unclear and I gave up" | disappointed |

## how will they experience it?

### journey touchpoints

```
discover → install → search → fetch → propose → research → distill → review
```

| touchpoint | experience |
|------------|------------|
| **discover** | hears about patenter from peer, docs, or search |
| **install** | npm install, rhachet roles init |
| **search** | `rhx patent.priors.search --query "..."` — sees relevant patents |
| **fetch** | `rhx patent.priors.fetch --exid US...` — reads full patent |
| **propose** | `rhx patent.propose` — route created, 0.idea.md opened |
| **research** | fills 0.idea.md, works through 3.1 stones |
| **distill** | sees patentability analysis in 3.2 stones |
| **review** | reviews blueprint, produces latex output |

### friction points

| friction | mitigation |
|----------|------------|
| API key setup required | clear docs, one-time setup |
| query refinement needed | suggestions on vague queries |
| stone workflow unfamiliar | mascot output explains each step |
| patent language unclear | briefs define terms |
| route feels long | clear progress indicators |

### smooth experience

1. `rhx patent.propose` — route created in 5 seconds
2. 0.idea.md opens with clear prompts
3. inventor describes idea in plain English
4. search runs automatically, results are relevant
5. favorable/adverse categorization makes sense
6. claims draft captures what's truly novel
7. latex output is attorney-ready
8. total time: hours, not weeks

---

## summary

**primary audience**: technical founders and indie inventors who need to research prior art and prepare patent materials before attorney engagement.

**why they care**: save $15-30k in attorney fees, understand their IP position, file with confidence.

**what they care about**: coverage, clarity, defensibility, cost, speed.

**how they'll experience it**: CLI-native workflow from search → fetch → route → output.

**success**: they say "I found patents I would have missed" and "my attorney said I came prepared."
