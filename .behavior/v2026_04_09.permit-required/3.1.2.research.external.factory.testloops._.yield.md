# research: test frameworks for rapid verification feedback

## project type analysis

| project type | fundamental experience to reproduce |
|--------------|-------------------------------------|
| **cli** | terminal output, command sequences, exit codes |
| **package** | api surface, return values, error conditions |

the permit system is primarily **CLI** (shell skills like permit.search, permit.fetch) with **package** aspects (thought route instantiation).

### extant test patterns

the repo already demonstrates CLI skill test patterns:

```typescript
// pattern: spawnSync wrapper for CLI test
const runSkill = (args: string[]): { stdout: string; stderr: string; exitCode: number } => {
  const result = spawnSync('bash', [skillPath, ...args], {
    encoding: 'utf-8',
    stdio: ['pipe', 'pipe', 'pipe'],
    timeout: 120000,
    env: process.env,
  });
  return {
    stdout: result.stdout ?? '',
    stderr: result.stderr ?? '',
    exitCode: result.status ?? 1,
  };
};
```

> "spawnSync spawns the shell synchronously and blocks Node.js until the child process exits or times out... preferred for test assertion simplicity" [1]

## tools for CLI experience reproduction

### 1. stdout/stderr capture

**convergence signal**: sources [1], [2], [3] agree that spawnSync with `stdio: ['pipe', 'pipe', 'pipe']` is the standard pattern.

> "For test of command-line tools, spawnSync is preferred because it blocks until completion, making assertions straightforward" [2]

> "The stdio option configures the pipes between parent and child process... 'pipe' creates a new pipe, giving access to stdout/stderr as strings" [1]

### 2. ANSI escape code stripping

> "strip-ansi removes ANSI escape codes from strings... essential for snapshot test of CLI output" [4]

> "jest-serializer-ansi-escapes converts ANSI escape codes to human-readable form in snapshots" [5]

**recommendation**: use strip-ansi for assertions, jest-serializer-ansi-escapes for readable snapshots.

### 3. stdin simulation

> "mock-stdin provides a way to simulate user input for interactive CLI tools in Jest tests" [6]

> "Integration test for interactive CLIs require careful stdin mock — the readable stream must emit data events at the right time" [7]

**note**: permit skills are non-interactive (args-based), so stdin simulation is low priority.

### 4. temp directory fixtures

the repo already has genTempDir from test-fns:

> "genTempDir creates an isolated temp directory with optional git init, fixture clone, and symlinks... auto-prunes after 7 days" [8]

```typescript
const tempDir = genTempDir({
  slug: 'permit-search-test',
  clone: fixturePath,
  git: true,
});
```

## snapshot test strategies

### stripping requirements

| dynamic content | approach |
|-----------------|----------|
| ANSI codes | strip-ansi |
| timestamps | replace with `[TIMESTAMP]` |
| file paths | replace with `[PATH]` |
| uuids | replace with `[UUID]` |

> "Effective snapshot test requires deterministic output — strip dates, random IDs, and environment-specific paths before the snapshot" [9]

> "Custom snapshot serializers transform output before comparison, enabling semantic equality rather than byte equality" [10]

### snapshot best practices

**convergence signal**: sources [9], [10], [11], [12] agree on these principles:

1. **small, focused snapshots** — one behavior per snapshot
2. **strip dynamic content** — dates, paths, IDs
3. **meaningful names** — describe what is captured
4. **review diffs carefully** — snapshots are symptoms, not the disease

> "Snapshots are best used for deterministic output like CLI help text, error messages, and structured data formats" [11]

> "Do not blindly update snapshots — the diff is the signal that behavior changed" [12]

## shell skill test frameworks

### bats-core

> "Bats is a TAP-compliant test framework for Bash executables... runs each test in a separate subshell for isolation" [13]

> "Bats provides @test, setup, teardown, and assertion utilities like run and assert_output" [14]

**note**: repo uses Jest with spawnSync instead of native bash test — this is correct for unified test runner across ts and sh.

### shellspec

> "ShellSpec is a BDD-style unit test framework for POSIX shell... supports mocks, stubs, and code coverage" [15]

**anti-pattern warning**: sources [13], [15] note that native shell test frameworks create a fragmented test suite when mixed with Jest.

> "Mixing shell test frameworks with Node test frameworks leads to split test reporting and CI complexity" [16]

**recommendation**: continue with Jest + spawnSync for unified test suite.

## exit code test pattern

**convergence signal**: sources [3], [20], [21] agree on exit code semantics:

| code | definition |
|------|------------|
| 0 | success |
| 1 | general error (malfunction) |
| 2 | misuse of command (constraint) |

> "Exit code 2 conventionally means incorrect usage or invalid arguments" [20]

> "In shell executables, explicit exit codes enable callers to distinguish recoverable from fatal errors" [21]

## recommended test pattern for permit skills

based on extant patterns and research:

```typescript
import { spawnSync } from 'child_process';
import * as path from 'path';
import { given, then, when, genTempDir } from 'test-fns';
import stripAnsi from 'strip-ansi';

describe('permit.search.sh', () => {
  const skillPath = path.join(__dirname, 'permit.search.sh');

  const runSkill = (args: string[]): { stdout: string; stderr: string; exitCode: number } => {
    const result = spawnSync('bash', [skillPath, ...args], {
      encoding: 'utf-8',
      stdio: ['pipe', 'pipe', 'pipe'],
      timeout: 120000,
    });
    return {
      stdout: stripAnsi(result.stdout ?? ''),
      stderr: stripAnsi(result.stderr ?? ''),
      exitCode: result.status ?? 1,
    };
  };

  given('[case1] valid address with permits', () => {
    when('[t0] search is invoked', () => {
      then('returns permits array', () => {
        const result = runSkill(['--address', '123 Main St', '--postal', '46220']);
        expect(result.exitCode).toBe(0);
        expect(result.stdout).toContain('permits:');
        expect(result.stdout).toMatchSnapshot();
      });
    });
  });

  given('[case2] invalid address', () => {
    when('[t0] search is invoked', () => {
      then('returns constraint error', () => {
        const result = runSkill(['--address', 'not a real address', '--postal', '00000']);
        expect(result.exitCode).toBe(2);
        expect(result.stderr).toContain('address not found');
      });
    });
  });
});
```

## anti-patterns

### avoid mock of child_process

> "Mock of child_process hides real integration issues — prefer real shell execution with isolated fixtures" [22]

### avoid skip on absent fixtures

> "Tests must fail fast with clear error when fixtures are absent, not skip silently" [23]

### avoid dynamic content in snapshots

> "Snapshots with timestamps or UUIDs will fail on every run — strip before capture" [9]
