# research: templates for permit system

## summary

this research identifies templates and patterns for the permit check and search system. the system requires CLI skill templates, thought route templates, legal research frameworks, and output format templates.

---

## 1. CLI skill templates

### 1.1 patenter skill pattern

the patenter role provides a reference implementation for permit skills.

> "patent.priors.search.sh: search for patents by keyword query... returns patents that match query (API order preserved), filters to patents with keywords in title, surfaces alerts for vague queries" [1]

> "patent.priors.fetch.sh: fetch patent document by application number from USPTO... retrieves full patent content for prior art analysis — claims, specification text, metadata" [1]

**convergence signal**: sources [1], [2], [3] agree on the standard skill header pattern:

```bash
#!/usr/bin/env bash
######################################################################
# .what = one line description
# .why  = why this skill exists
# usage: skill.sh --arg value
# guarantee: what the skill guarantees
######################################################################
```

### 1.2 treestruct output pattern

> "Treestruct format uses turtle vibes with eagle mascot header, shell root names the command, branches show hierarchical data" [2]

```
🦅 lets soar and see,

🏛️ permit.search --address "123 Main St" --postal 46220
   ├─ permits: 5 found
   │  ├─ INDY-2019-00012345
   │  │  ├─ type: electrical
   │  │  ├─ status: 🟢 issued
   │  │  └─ date: 2019-03-15
   └─ source: Indianapolis-Marion County
```

### 1.3 exit code conventions

**convergence signal**: sources [3], [4], [5] agree on exit code semantics:

| code | definition |
|------|------------|
| 0 | success |
| 1 | malfunction (external error) |
| 2 | constraint (user must fix) |

> "Exit code 1 vs 2 tells caller whether to retry or fix — constraint errors (2) indicate user action required before retry" [3]

## 2. thought route templates

### 2.1 stone progression pattern

the behavior route demonstrates a structured research workflow:

> "Stone workflow: 0.query → 1.vision → 3.1.research.* → 3.2.distill → 3.3.blueprint → 5.deliver" [6]

**for permit determination route**:

| stone | purpose |
|-------|---------|
| 0.query | postal code + work description |
| 1.vision | determine permit requirement |
| 3.1.research.jurisdiction | identify state/local code adoption |
| 3.1.research.baseline | extract R105.1 (permits required) |
| 3.1.research.exemptions | extract R105.2 (exempt work) |
| 3.2.distill.citations | compile verbatim quotes |
| 3.3.blueprint.determination | structure the proof chain |
| 5.1.deliver.diagnosis | thorough markdown analysis |
| 5.3.deliver.coverpage | treestruct one-page summary |

### 2.2 query stone template

> "0.query stone is pre-filled with postal and work so user can review and refine before research" [7]

```markdown
# permit.check.required query

## postal
46220

## work
electrical panel upgrade from 100A to 200A service
```

## 3. legal research templates

### 3.1 IRAC framework

**convergence signal**: sources [8], [9], [10], [11] agree on IRAC as the standard legal analysis framework:

> "IRAC (Issue, Rule, Application, Conclusion) is the cornerstone of legal analysis... provides a structured approach that ensures thorough and organized analysis" [8]

> "The IRAC method provides a reliable framework for legal write and analysis used by law students and attorneys" [9]

> "Issue: What is the legal question? Rule: What law applies? Application: How does the law apply? Conclusion: What is the result?" [10]

### 3.2 permit determination proof chain

the permit determination follows an IRAC-like structure:

> "Prove permits ARE required first (R105.1), then prove work is NOT exempt (R105.2) — if not exempt, permit required" [12]

**proof chain template**:

1. **BASELINE**: cite R105.1 (work that requires permits)
2. **EXEMPTIONS**: cite R105.2 (work exempt from permits)
3. **ANALYSIS**: map work description to code language
4. **CONCLUSION**: required / not_required / conditional / unclear

### 3.3 code citation format

> "Code citations must include exact quotes with section numbers, not paraphrases... misquotes of code are grounds for dismissal" [13]

```markdown
### baseline (IRC R105.1)

> "Any owner or owner's authorized agent who intends to construct, enlarge, alter,
> repair, move, demolish or change the occupancy of a structure, or to
> erect, install, enlarge, alter, repair, remove, convert or replace any electrical,
> gas, mechanical or pipe system... shall first make application to the
> official and obtain the required permit." [14]
```

## 5. anti-patterns

### 5.1 avoid paraphrase of code

> "Never paraphrase legal code — always use verbatim quotes with exact section numbers" [13]

> "Misquotes or paraphrase of statutes can lead to incorrect legal conclusions" [17]

### 5.2 avoid single-source conclusions

> "Legal conclusions should be supported by multiple sources — statute, regulation, and authoritative commentary" [18]

### 5.3 avoid implicit jurisdiction

> "Always explicitly state which code version applies — state adoptions vary and may include amendments" [19]

### 5.4 avoid overconfident determinations

> "When code language is ambiguous, the correct output is 'unclear' with questions for the permit office, not a forced conclusion" [12]
