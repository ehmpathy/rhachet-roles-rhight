## summary

this research identifies blockers that could halt or slow permit system development. the primary blocker is **Accela API access** — Indianapolis uses Accela for permits, but API access requires agency partnership approval which is typically denied.

## access and credentials

### assessment: BLOCKER — Accela API access denied

**the blocker**: Indianapolis uses Accela for permits. Accela exposes a REST API, but access requires agency registration and approval.

from `howto.indianapolis-permit-api-search.md`:

> "you need agency registration. this means:
> 1. register app at developer.accela.com
> 2. request access to INDY agency environment
> 3. Indianapolis DBNS must approve your app
> 4. receive client_id and client_secret
>
> status: most requests are denied unless you're a government contractor or civic app partner."

### impact

| approach | status | blocker |
|----------|--------|---------|
| Accela API | blocked | agency approval required |
| Open Indy Data Portal | no permit data | permits not exposed |
| web scrape | available | fragile, ToS concerns |
| manual portal | available | does not scale |

### resolution paths

1. **apply for Accela API access** (low probability of approval)
   - contact: permitquestions@indy.gov
   - timeline: unknown
   - outcome: likely denied

2. **use web scrape as primary** (recommended for MVP)
   - target: `https://aca-prod.accela.com/INDY/Cap/CapHome.aspx?module=Permits`
   - tools: Playwright (per `3.1.2.research.external.factory.oss.levers._.yield.md`)
   - risk: selectors change when Accela updates UI
   - mitigation: cache results, run scrape in controlled batches

3. **manual portal as fallback**
   - for one-off lookups
   - document in skill usage

### credentials needed (if scrape approach)

no credentials needed for public portal scrape. if we later obtain API access:

```
ACCELA_INDY_CLIENT_ID
ACCELA_INDY_CLIENT_SECRET
```

## other blockers

### 1. portal scrape fragility

**blocker level**: medium

Accela portal UI changes periodically. selector-based scrape will break.

**mitigation**:
- cache results aggressively
- version selectors in code
- monitor for breakage in CI
- document selector update procedure

### 2. rate limits and IP blocks

**blocker level**: low

too many requests may trigger blocks.

**mitigation**:
- use p-queue for concurrency control (per oss.levers research)
- add retry with backoff (reuse patent.priors pattern)
- cache results to minimize requests

### 3. absence of bulk export

**blocker level**: medium

portal does not support bulk export. each permit requires individual lookup.

**mitigation**:
- cache results locally
- batch lookups at off-peak hours
- consider scope: do we need bulk search or just individual lookups?

### 4. terms of service concerns

**blocker level**: unknown

scrape may violate Accela/Indianapolis portal ToS.

**mitigation**:
- review ToS before production use
- consider official API application even if likely denied
- document legal review needed before production

### 5. permit data structure variance

**blocker level**: low

different permit types may have different data shapes.

**mitigation**:
- start with electrical permits (per wish)
- extend to other types incrementally
- use flexible parse with fallbacks

| portal scrape fragility | medium | acceptable | cache + version selectors |

## recommended approach

1. **use web scrape for MVP** — Accela API unlikely to be approved
2. **cache aggressively** — minimize portal requests
3. **use fixtures for tests** — deterministic verification
4. **version selectors** — track breakage in CI
5. **document ToS review need** — before production use
6. **apply for API access in parallel** — low probability but worth attempt
