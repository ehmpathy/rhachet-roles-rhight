# research: internal test codepath patterns

## summary

this research documents extant test codepath patterns in the rhight repo that are relevant to permit skill tests. patterns are marked [REUSE], [EXTEND], or [REPLACE] based on fit.

## 1. integration test structure [REUSE]

from `patent.priors.search.integration.test.ts`:

```typescript
import { spawnSync } from 'child_process';
import * as path from 'path';
import { given, then, when } from 'test-fns';

describe('skill-name.sh', () => {
  const scriptPath = path.join(__dirname, 'skill-name.sh');

  const runSkill = (input: {
    skillArgs: string[];
  }): { stdout: string; stderr: string; exitCode: number } => {
    const result = spawnSync('bash', [scriptPath, ...input.skillArgs], {
      encoding: 'utf-8' as const,
      stdio: ['pipe', 'pipe', 'pipe'],
      timeout: 60000,
      env: process.env,
    });
    return {
      stdout: result.stdout ?? '',
      stderr: result.stderr ?? '',
      exitCode: result.status ?? 1,
    };
  };

  // test cases...
});
```

**fit**: direct reuse for permit.search and permit.fetch integration tests.

```typescript
given('[case8] deterministic cached flow (uses fixture)', () => {
  const knownExid = '19399196';
```

## 7. exit code semantics in tests [REUSE]

```typescript
given('[case2] query too short', () => {

      const result = runSearch({ searchArgs: ['--query', 'ab'] });

      // error messages go to stderr

    });
  });

```

| code | definition | test assertion |
|------|------------|----------------|
| 0 | success | `expect(result.exitCode).toBe(0)` |
| 1 | malfunction | `expect(result.exitCode).toBe(1)` |
| 2 | constraint | `expect(result.exitCode).toBe(2)` |

**fit**: direct reuse. permit skills follow same exit code semantics.

```typescript
  const scriptPath = path.join(__dirname, 'asStatusEmoji.sh');
```
