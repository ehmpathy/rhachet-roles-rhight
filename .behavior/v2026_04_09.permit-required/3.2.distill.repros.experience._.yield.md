## critical paths

| critical path | description | why critical |
|---------------|-------------|--------------|
| happy: check required | postal + work → determination with proof chain | core value prop |
| happy: search permits | address → list of permits | property due diligence |
| happy: fetch permit | ID → full record with inspections | verification of contractor claims |
| error: portal unavailable | graceful error with retry hint | user not left confused |
| error: ambiguous work | ask for clarification, not forced guess | wrong determination is worse than "unclear" |

the "happy: check required" path is the money shot. if this fails or has friction, the product fails.

---

## ergonomics review

| journey | input ergonomics | output ergonomics | friction notes |
|---------|------------------|-------------------|----------------|
| permit.check.required | natural — postal + work in plain language | natural — proof chain is citable | work interpretation may need clarification |
| permit.search | natural — address as user would type | natural — permits list is scannable | pagination needs clear cursor UX |
| permit.fetch | awkward — permit ID format specific | natural — full record with timeline | ID discovery relies on prior search |

### input friction: permit.fetch

permit IDs like `INDY-2019-00012345` are not user-friendly. users discover these via `permit.search` first. this is acceptable — the journey is: search → pick ID from results → fetch details.

### output friction: work interpretation

when work description is ambiguous, the system should ask for clarification rather than guess. this is correct ergonomics — a wrong determination is worse than "unclear".

## reproduction feasibility

### permit.check.required (thought route)

**feasible**: tests can:
1. invoke skill with postal + work
2. drive route stones to completion
3. assert on yield files (5.1.deliver.diagnosis.yield.md, 5.3.deliver.coverpage.yield.md)
4. snapshot output

**utilities available**:
- spawnSync for skill invocation
- fs for yield file assertions
- test-fns for BDD structure

**setup required**:
- fixture jurisdiction data
- fixture code sections (IRC R105.1, R105.2)

### permit.search (portal scrape)

**feasible with fixtures**: tests can:
1. mock portal response via cached HTML
2. invoke skill with address
3. assert on parsed permits
4. snapshot output

**utilities available**:
- Playwright for live scrape (if needed)
- cached fixtures for deterministic tests

**setup required**:
- cached HTML fixtures for Indianapolis Accela portal
- selector version files

### permit.fetch (portal scrape)

**feasible with fixtures**: same as permit.search — cached fixtures enable deterministic tests.
