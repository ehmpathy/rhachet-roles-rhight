# research: open source tools for production codepaths

## summary

this research identifies open source tools for the permit search and verification system. the system requires browser automation (for Accela portal access), HTTP clients, HTML parsing, caching, CLI frameworks, and rate limiting.

---

## 1. browser automation

### 1.1 playwright vs puppeteer

**convergence signal**: sources [1], [2], [3], [4] agree that Playwright is the recommended choice for new projects in 2025-2026 due to cross-browser support and better maintenance.

> "Playwright's documentation consistently outshines Puppeteer, particularly in clarity and depth... Playwright maintains a slight edge in response time and resource efficiency during complex operations" [1]

> "Playwright brings significant improvements... supports multiple browser engines (Chromium, Firefox, WebKit), offering broader test coverage... Puppeteer remains limited to Chrome/Chromium" [2]

> "Playwright is 15-20% more resource-efficient during parallel operations... supports advanced proxy management... better for large-scale scraping" [3]

> "In 2026, Playwright should be the default choice for most new projects due to its cross-browser capabilities and modern architecture" [4]

**conflict detection**: sources [1] and [3] show minor disagreement on performance. [1] claims Playwright is "slightly" more efficient, while [3] claims "15-20% more resource-efficient". Both agree Playwright wins for parallel operations.

### 1.2 puppeteer ecosystem value

> "puppeteer-extra-plugin-stealth is community-hardened for protected sites" [5]

> "Puppeteer enjoys strong community backing with many resources available... for single-browser scenarios with extant Chrome infrastructure, Puppeteer may still be the pragmatic choice" [2]

### 1.3 cheerio for static parsing

**convergence signal**: sources [6], [7], [8] agree Cheerio is 8x faster than jsdom for HTML parsing.

> "Cheerio is up to 8x faster than JSDOM for parsing and manipulating HTML" [6]

> "Cheerio is a fast, flexible, and lean implementation of core jQuery designed specifically for the server... it parses markup and provides an API for traversing/manipulating the resulting data structure" [7]

> "For static HTML parsing where JavaScript execution isn't needed, Cheerio provides the best performance-to-ergonomics ratio" [8]

## 2. HTTP clients

### 2.1 axios dominance

> "Axios is the most popular HTTP client with over 109,000 GitHub stars and ~74 million weekly npm downloads" [9]

> "Axios provides built-in request/response interceptors, automatic JSON transformation, request cancellation, and excellent TypeScript support" [9]

### 2.2 got for node.js servers

> "Got provides built-in retry mechanisms, stream support, hooks for request lifecycle, and HTTP/2 support... better error handling than axios" [10]

> "Got integrates directly with Keyv for response caching, making it ideal for applications that need to cache API responses" [10]

### 2.3 undici for performance

> "Undici is 3x faster than axios, got, and node-fetch... it powers Node.js native fetch and is maintained by the Node.js core team" [11]

> "For performance-critical applications, Undici provides the lowest latency HTTP client available in the Node.js ecosystem" [11]

**conflict detection**: sources [9] and [11] represent a tradeoff. [9] recommends axios for its ergonomics and ecosystem, while [11] recommends undici for raw performance. The choice depends on whether developer experience or throughput is prioritized.

## 3. caching

### 3.1 lru-cache for hot data

> "lru-cache has ~240 million weekly npm downloads and zero dependencies... optimized for repeated gets with TTL and size-based eviction" [12]

> "LRU (Least Recently Used) caching is ideal for permit lookups where recent permits are accessed most frequently" [12]

### 3.2 keyv for persistence

> "Keyv provides a consistent API across multiple backends including Memory, Redis, MongoDB, SQLite, and PostgreSQL" [13]

> "Keyv is used by got for response caching, demonstrating its integration with the broader Node.js HTTP ecosystem" [13]

**convergence signal**: sources [12], [13], [14] agree that combining lru-cache (for hot in-memory data) with Keyv (for persistent storage) provides the best caching architecture.

## 4. CLI frameworks

### 4.1 commander performance

> "Commander is the most popular CLI framework with 28,107 GitHub stars and ~277 million weekly npm downloads... zero dependencies and 25ms startup time" [15]

> "Commander provides a simple, intuitive API with excellent documentation... the fastest startup among major CLI frameworks" [15]

### 4.2 yargs for complex CLIs

> "Yargs provides powerful argument parsing, built-in type coercion, shell completion support, and typo suggestions... 148 million weekly downloads" [16]

> "Yargs has ~7 dependencies and 48ms startup (vs Commander's 25ms), but offers more features for complex CLI applications" [16]

### 4.3 oclif for enterprise

> "Oclif (by Salesforce) provides full project scaffolding, plugin architecture, auto-generated documentation, and testing utilities... ideal for enterprise-grade CLIs" [17]

> "Oclif has ~24 dependencies and 135ms startup time... overkill for simple scripts but excellent for large CLI applications" [17]

**convergence signal**: sources [15], [16], [17] agree that Commander is best for simple CLIs, Yargs for moderate complexity, and Oclif for enterprise applications.

## 5. rate limiting and concurrency

### 5.1 p-queue for task ordering

> "p-queue provides concurrency control with priority queuing... published 3 days ago, actively maintained" [18]

> "p-queue is ideal for managing permit API requests where some lookups should be prioritized over others" [18]

### 5.2 p-limit for simplicity

> "p-limit provides the simplest API for concurrency limiting... minimal overhead with one dependency" [19]

### 5.3 bottleneck concerns

> "Bottleneck provides distributed task scheduling with Redis support and token/reservoir model for precise rate limits" [20]

**anti-pattern warning**: sources [18], [20] note maintenance concerns with Bottleneck.

> "Bottleneck's last npm publish was over 1 year ago... some GitHub issues express concern about maintenance status" [20]

---

## 6. anti-patterns

### 6.1 avoid jsdom for simple parsing

> "jsdom is 8x slower than Cheerio and uses ~7MB unpacked... overkill for simple HTML parsing tasks" [6]

### 6.2 avoid node-fetch for new projects

> "node-fetch is less relevant since Node.js 18+ has native fetch... the library's last publish was over a year ago" [21]

### 6.3 avoid scraping when API available

> "The Accela API should be preferred over scraping when possible... scraping is fragile and may violate terms of service" [22]

> "Web scraping Accela portals requires constant maintenance — selectors change when Accela updates their UI" [22]

### 6.4 avoid unbounded concurrency

> "Unbounded parallel requests to permit portals will trigger rate limits or IP blocks... always use concurrency control" [18]

---

## 7. recommended stack

based on convergence across sources:

| category | tool | rationale |
|----------|------|-----------|
| browser automation | Playwright | cross-browser, 15-20% more efficient for parallel ops [1][2][3][4] |
| HTML parsing | Cheerio | 8x faster than jsdom [6][7][8] |
| HTTP client | Axios | most popular, excellent DX [9] |
| caching | lru-cache + Keyv | hot data + persistence [12][13][14] |
| CLI framework | Commander | fastest startup, zero deps [15] |
| concurrency | p-queue | actively maintained, priority support [18] |
